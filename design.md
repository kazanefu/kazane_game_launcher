# 設計

## 概要

Kazane Game Launcherはkazane(私)が開発したゲームを検索/インストール/起動/アンインストール/アップデート/説明の表示を行うためのランチャーです。

売りにする機能は以下の通りです。
- kazaneが開発したゲームを簡単に管理できる
- GitHubのREADME.mdをmarkdown表示することで、ゲームの説明をわかりやすく表示できる
- GitHubのリポジトリへのリンクを表示することで、ユーザーがゲームのソースコードや最新の情報を簡単に確認できる
  - 既に別の方法でインストール済みのアプリケーションをライブラリに追加する機能を実装することで、ユーザーが既にインストールしているゲームも管理できるようにする
- 軽量で高速なランチャー

## 画面構成

- 検索画面
  - ゲームの検索及びインストールを行う画面
  - 検索結果の表示
  - 説明の表示(GitHubのREADME.mdをmarkdown表示)
  - GitHubのリポジトリへのリンク
- ライブラリ画面
  - インストール済みゲームの表示
  - ゲームの起動/アンインストール/アップデートを行う画面
  - 説明の表示(GitHubのREADME.mdをmarkdown表示
  - 既に別の方法でインストール済みのアプリケーションをライブラリに追加する機能
  - ゲームのインストール先の変更
  - インストール場所をexplorerで開く機能
- 設定画面(実装の優先度は低い(最初は後述のlauncher/settings.jsonを直接編集してもらう形にする予定))
  - ゲームのインストール先の変更
  - その他ランチャーの設定
  - テーマを変更(ダークモード/ライトモード)
  - ランチャーのUpdate

## 技術スタック(予定)

- GUI: eframe egui
- HTTPクライアント: reqwest
- Markdown表示: egui_commonmark
- データの保存: serde + json + ファイルシステム
- GitHub APIを使用してゲームのリストを取得し、インストールやアップデートの際に最新のリリース情報を取得する
- 非同期ランタイム: tokio
- その他のユーティリティクレート: zip, directories, etc.

## データ構造

### ローカル (local/game_data.json, launcher/settings.json, games/) (初回起動時に生成)

注: ランチャーの実行ファイルからの相対パスで記載

local/game_data.json

- インストール済みゲームの情報のリスト
  - ゲーム情報
    - ゲーム名
    - バージョン
    - インストール先のパス
    - GitHubのリポジトリURL
    - "installed"フラグ

launcher/settings.json

- ランチャーの設定
  - ゲームのインストール先のパス
  - テーマ(ダークモード/ライトモード)
  - その他ランチャーの設定

games/
- デフォルトのゲームのインストール先
- インストールされたゲームの実行ファイルやその他の関連ファイルが配置されるディレクトリ

### リモート(GitHub: data/game_list.json　リポジトリのルートに配置, launcher/release.json各ゲームのリポジトリルートに配置)

data/game_list.json

- ゲームのリスト
  - ゲーム情報
    - ゲーム名
    - GitHubのリポジトリURL

launcher/release.json

- ゲームのリリース情報
  - バージョン
  - Github releaseのURL(zipや.exeのどれをインストールするのかを判断するため)
  - ファイルタイプ(zipか.exeか)
  - zipの場合
    - 解凍後の実行ファイルのパス
  - .exeの場合
    - そのままインストール

## インストールのフロー

1. ユーザーが検索画面でゲームを検索(とりあえずはゲーム名で完全一致検索, 将来的には部分一致やREADMEの内容で検索できるようにするかも)
2. GitHub上においてあるdata/game_list.jsonからゲームのリストを取得し、検索結果を表示(ゲーム名のみを表示)
3. ゲーム名をクリックすると、ゲームの説明とインストールボタンとGitHubのリポジトリへのリンクを表示
4. ユーザーがゲームを選択し、インストールボタンをクリック
5. 各ゲームのリポジトリルートに配置されているlauncher/release.jsonから最新のリリース情報を取得し、インストールするファイルをダウンロード(zipか.exe)(games/(or ユーザーが指定したディレクトリ(launcher/settings.jsonを参照))に直接インストール)
6. zipの場合は解凍
7. 実行ファイルのパスとかlocal/game_data.jsonに追加すべき情報を保存
8. インストール完了の通知を表示(インストール中はプログレスバーを表示)
9. 更新されたlocal/game_data.jsonをもとにライブラリ画面の表示を更新

## 起動のフロー

1. ユーザーがライブラリ画面でゲームを選択
2. ゲームの説明と起動/アンインストール/アップデートボタンを表示
3. ユーザーが起動ボタンをクリック
4. local/game_data.jsonからゲームの実行ファイルのパスを取得し、そのパスのファイルを起動
5. ゲームが正常に起動したかどうかを確認(プロセスが正常に開始されたかどうかを確認)
6. 起動に成功した場合は、ゲームが起動していることをユーザーに通知し、起動に失敗した場合はエラーメッセージを表示
7. ゲームが正常に起動した場合は、ゲームのプロセスが終了するまで待機し、ゲームが終了したことをユーザーに通知(ゲームが実行中のときは起動ボタンが実行中/終了ボタンに変わる)
8. 実行中/終了ボタンをクリックすると、ゲームのプロセスを強制終了する機能も実装する予定
9. ゲームのプロセスが終了したら起動ボタンに戻る

## アンインストールのフロー

1. ユーザーがライブラリ画面でゲームを選択
2. ゲームの説明と起動/アンインストール/アップデートボタンを表示
3. ユーザーがアンインストールボタンをクリック
4. インストールされたゲームのファイルを削除(games/(or ユーザーが指定したディレクトリ(launcher/settings.jsonを参照)から削除(`std::fs::remove_dir_all()`)
5. local/game_data.jsonから該当するゲームのinstalledフラグをfalseにするか、該当するゲームの情報を削除
6. アンインストール完了の通知を表示
7. 更新されたlocal/game_data.jsonをもとにライブラリ画面の表示を更新

## アップデートのフロー

1. ユーザーがライブラリ画面でゲームを選択
2. ゲームの説明と起動/アンインストール/アップデートボタンを表示
3. ユーザーがアップデートボタンをクリック
4. local/game_data.jsonを参照してから各ゲームのリポジトリルートに配置されているlauncher/release.jsonから最新のリリース情報を取得し、現在インストールされているバージョンと比較して、アップデートが必要かどうかを判断
5. アップデートが必要な場合は、インストールのフローの5以降を実行してアップデートを行う
6. アップデートが完了したら、local/game_data.jsonの該当するゲームのバージョンを更新
7. アップデート完了の通知を表示
8. 更新されたlocal/game_data.jsonをもとにライブラリ画面の表示を更新

## explorerで開く機能のフロー

1. ユーザーがライブラリ画面でゲームを選択
2. ゲームの説明と起動/アンインストール/アップデートボタンを表示
3. ユーザーが「インストール場所をexplorerで開く」ボタンをクリック
4. local/game_data.jsonからゲームのインストール先のパスを取得し、そのパスをexplorerで開く(`std::process::Command::new("explorer").arg(path).spawn()`)
5. explorerが正常に起動したかどうかを確認(プロセスが正常に開始されたかどうかを確認)
6. 起動に成功した場合は、explorerが起動していることをユーザーに通知し、起動に失敗した場合はエラーメッセージを表示

## エラー処理

- エラー処理は基本的にユーザーにわかりやすいエラーメッセージを表示する形で行う
- パニックしないように、エラーが発生した場合は適切に処理して、ユーザーに通知する
- エラーレベル
  - recoverable error: ユーザーが操作をやり直すことで解決できるエラー(例: ネットワークエラーなど)
  - fatal error: ユーザーが操作をやり直しても解決できないエラー(例: ファイルの読み書きエラーなど)
- recoverable errorが発生した場合は、ユーザーにエラーメッセージを表示し、操作をやり直すように促す
- fatal errorが発生した場合は、ユーザーにエラーメッセージを表示し、必要に応じてランチャーを再起動するように促す

## ランチャーのインストール/アンインストール/アップデート

- インストーラを作成して、ユーザーが簡単にランチャーをインストールできるようにする予定
- インストーラは、ランチャーの実行ファイルをユーザーの指定したディレクトリにコピーし、必要な初期設定ファイル(local/game_data.json, launcher/settings.json)を生成する機能を持つ予定
- インストーラは、ユーザーがランチャーをアンインストールするための機能も持つ予定(ランチャーの実行ファイルと初期設定ファイルを削除する機能)
- インストーラは、ユーザーがランチャーをアップデートするための機能も持つ予定(新しいバージョンのランチャーの実行ファイルをユーザーの指定したディレクトリにコピーし、必要な初期設定ファイル(local/game_data.json, launcher/settings.json)を更新する機能)

## 大まかなコードの設計の方針

- コードはモジュールごとに分割して、各モジュールが特定の機能を担当するようにする予定
- コードの再利用性/可読性/保守性を高めるため、OOP的な実装にするが, `&dyn Trait`のようなトレイトオブジェクトを多用するのは避ける(パフォーマンスの観点から)
- 優先順位 パフォーマンス > 可読性, 保守性 > 開発速度
- テストをちゃんと書いて、コードの品質を保つようにする予定
- 非同期処理を適切に使用して、ユーザーの操作に対して迅速に反応できるようにする予定(例: ゲームのインストールやアップデートの際のダウンロード処理など)
- エラー処理を適切に行い、ユーザーにわかりやすいエラーメッセージを表示するようにする予定
- コードのドキュメントをしっかり書いて、コードの理解を助けるようにする予定
- コードのリファクタリングを定期的に行い、コードの品質を保つようにする予定
- ファイルは細かく分割して、1ファイルあたりの行数が多くなりすぎないようにする予定
- コードの命名規則は、Rustの一般的な命名規則に従う予定(例: スネークケースで関数や変数を命名する、パスカルケースで構造体や列挙型を命名するなど)
- clippyを使用して、コードの品質を保つようにする予定
- warningsをすべて解消して、コードの品質を保つようにする予定

## 付録: 追加仕様とサンプル
以下は設計に追記・明確化した項目です。セキュリティ、信頼性、UI/運用面、テスト、Windows固有挙動などを扱います。

### JSON スキーマ(例)
- data/game_list.json (リポジトリルート)
```json
{
  "games": [
    {
      "id": "sample-game",
      "name": "Sample Game",
      "repo": "https://github.com/owner/sample-game",
      "description": "短い説明(オプション)",
      "tags": ["puzzle", "arcade"]
    }
  ]
}
```

- launcher/release.json (各ゲームのリポジトリルート)
```json
{
  "latest": {
    "version": "1.2.3",
    "assets": [
      {
        "name": "sample-game-win.zip",
        "url": "https://github.com/owner/sample-game/releases/download/v1.2.3/sample-game-win.zip",
        "type": "zip",
        "sha256": "<hex-sha256>",
        "entry_point": "sample-game.exe"
      }
    ],
    "channel": "stable"
  }
}
```

- local/game_data.json (ランチャーローカル)
```json
{
  "installed": [
    {
      "id": "sample-game",
      "name": "Sample Game",
      "version": "1.2.3",
      "install_path": "C\\\\Games\\\\sample-game",
      "repo": "https://github.com/owner/sample-game",
      "installed": true,
      "last_checked": "2026-05-08T00:00:00Z"
    }
  ]
}
```

### ダウンロードの検証
- 常にHTTPSを使用する。
- release.jsonにsha256を含め、ダウンロード後にハッシュ検証を行う。
- 将来的に署名(公開鍵)検証の導入を検討する。

### インストールの原子性とロールバック
- インストールは一時ディレクトリにダウンロード→検証→展開→最終インストール先へ原子的に移動(rename)する。
- 失敗時は一時ディレクトリを削除して状態をロールバックし、local/game_data.jsonは成功時にのみ更新する。
- 更新は一時ファイルで行い、最後にファイルロックを取得してlocal/game_data.jsonへ置換する。

### 同期とファイルロック
- local/game_data.jsonへのアクセスは排他ロックを使用(例: ファイルロックまたはアプリ内Mutex+ファイル単位の排他)する。
- 外部で編集された場合の競合解決手順を設計(最終更新時刻の比較、ユーザーに選択肢を提示)

### Windows 固有考慮事項
- UACが必要な場所(Program Files等)へインストールする際は権限エラーをハンドリングし、ユーザーにインストール先変更を促す。
- パスは常に\を使用し、パス結合には標準ライブラリ(API)を使う。
- explorer起動はCommand::new("explorer").arg(path).spawn()だが、パス検証を厳密に行いコマンドインジェクションを防ぐ。
- プロセス起動時はフルパスを指定し、引数や環境変数は適切にエスケープ/検証する。

### エラー分類と再試行方針
- recoverable: ネットワーク一時障害等 → 指数バックオフで最大N回リトライ。ユーザーにキャンセル可能にする。
- fatal: ファイルシステムの恒久的エラー等 → 明確なメッセージと復旧手順を提示。

### UI/UX の追加仕様
- ダウンロード/展開中の進捗バーと速度表示
- キャンセルボタン（押下時はダウンロード停止と一時ファイルのクリーンアップ）
- 操作ログ出力(ユーザーフレンドリな操作履歴)と詳細ログ(問題時の診断用)
- 通知(トースト)の仕様: 成功/失敗/完了/中断

### セキュリティ対策
- 入力されたパスやURLは正規化・検証してディレクトリ横移動などを防ぐ。
- 外部コマンドを通す処理は最小限にし、可能な限り標準APIを利用して直接プロセスを起動する。
- ダウンロード元は信頼できるURLのみを許可(リポジトリのrelease.json経由)

### テスト/CI
- ユニットテスト: JSONパーサ、ハッシュ検証、パス正規化など
- 統合テスト: 一時ディレクトリを使ったインストールフローのエンドツーエンドテスト
- CI: push/PRでcargo testを実行、clippy/formatチェックを含める

### バージョニングとアップデート戦略
- release.jsonにversionとsha256を含め、既存バージョンと比較して差分(パッチ)配布かフル配布かを判断する。
- ランチャー自身の更新は自己更新/外部アップデータどちらかを検討。自動更新はユーザー承認を求める。

- design.mdにサンプルと要点を追記済み。詳細なフローはdocs/下に分割して保持する予定。
